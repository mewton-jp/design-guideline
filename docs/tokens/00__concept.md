# Design Tokens – Concept

## このドキュメントについて

このドキュメントは、  
本プロジェクトにおける **Design Token の考え方そのもの**を定義する。

ここで定義されるのは、

- 具体的な色
- 数値
- コンポーネント仕様

ではない。

**「なぜこの構造でトークンを設計するのか」**  
という前提と制約を明文化することを目的とする。

---

## 基本姿勢

この Design Token 群は、

- 見た目を統一するため
- ブランド感を固定するため
- デザインを自動化するため

に存在しない。

Design Token は、

> **判断を簡略化し、  
> 思考と実装の間に翻訳層を作るための仕組み**

として扱う。

---

## 従来のトークン設計を採用しない理由

一般的な Design Token 設計で用いられる、

- `primary / secondary`
- `brand color`
- `success / error`

といった分類は、本プロジェクトの思想と整合しない。

### Primary / Secondary を採用しない理由

- 「主役 / 脇役」という固定的な序列を生む
- 文脈によって前後する役割を表現できない
- ノートやマインドマップの構造と相性が悪い

本プロジェクトでは、  
**役割は固定されず、状況によって変化する**。

---

### Brand Color を定義しない理由

- 「常に前に出る色」を作ることは、
  「エゴを消す」「基盤に徹する」という原則に反する
- 色は自己主張ではなく、状態や距離感を示すために使われる

---

### Success / Error を採用しない理由

- 正解 / 間違いという二項対立は、
  思考のプロセス（Metaphor: Note）と相容れない
- 判断や評価を断言しない、という姿勢と矛盾する

---

## レイヤ構造

Design Token は、以下の 3 層構造で設計される。

---

### 1. Base Layer（基底層）

Base Layer は、  
意味を持たない **物理的・視覚的素材**の集合である。

- カラーパレット
- タイポグラフィの候補
- スケール値
- 単位系

ここにある値は、

- 直接使用しない
- 意味を付与しない
- 文脈を持たない

#### 例

- `palette.gray.500`
- `palette.chroma.cyan.600`
- `font.sans.default`
- `font.mono.code`

Base Layer は **辞書**であり、  
判断は含まれない。

---

### 2. Metaphor Layer（意味の源泉）

Metaphor Layer は、  
「どのような認知状態・文脈で使われているか」を定義する。

- note
- mindmap
- sandbox
- attached_image
- external_window

ここには、

- なぜそう振る舞うのか
- どのような態度で情報に向き合うのか

といった **思想・構造**が存在する。

Metaphor Layer は Token ではない。  
意味の出発点である。

---

### 3. Semantic Layer（翻訳層）

Semantic Layer は、  
**特定のメタファーの中でのみ意味を持つ語彙**である。

Semantic Token は、  
グローバルに共通化しない。

#### 重要な原則

> Semantic Token は  
> **メタファーごとに定義し直される**

---

#### 例（Note メタファー）

- `note.ink.loud`
- `note.ink.regular`
- `note.ink.quiet`
- `note.surface.paper`
- `note.spacing.flow`

これらは **Note の文脈でのみ意味を持つ**。

同じ語が、  
Mindmap や Sandbox で使われることは前提としない。

---

## なぜ Semantic を共通化しないのか

Semantic Token は「意味」を表す。

しかし本プロジェクトでは、

- メタファーが変わる
- 認知状態が変わる
- 同じ強度や色が、異なる意味を持つ

ため、

> **Semantic をグローバルに定義すると、  
> 必ず意味が破綻する**

`ink-loud` が Note では意味を持ち、  
Mindmap では意味不明になるのは、  
設計の失敗ではない。

**正しくレイヤが分離されている証拠**である。

---

## Token の役割

Design Token は、思想を語らない。

- 性格を表現しない
- メタファーを直接表さない
- 意味を内包しない

Token が持つのは、

- 強度
- 密度
- 距離
- 切り替えの有無

といった **思想の影**だけである。

---

## 禁止事項

以下の設計は行わない。

- Metaphor 名を Token 名にする
- Semantic Token を全体共通語彙として定義する
- 色や数値に直接意味を書き込む
- 「使いやすさ」のために思想を省略する

---

## 要約

> Design Token は、  
> 思想を固定するためのものではない。  
>  
> 思想を壊さずに、  
> 実装へ翻訳するための装置である。
